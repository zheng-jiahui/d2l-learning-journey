## 理论题

### 1. 线性回归和逻辑回归的区别是什么？损失函数分别是什么？

| | **线性回归** | **逻辑回归** |
|---|---|---|
| **任务类型** | 回归（预测连续值） | 二分类（预测概率） |
| **输出** | 无约束实数值 | 通过 Sigmoid 压缩到 (0,1) |
| **损失函数** | 均方误差（MSE） | 二元交叉熵（Binary Cross Entropy） |
| **假设** | 因变量服从高斯分布 | 因变量服从伯努利分布 |
| **决策边界** | 线性超平面 | 线性超平面（在 logit 空间） |

**损失函数形式**：
- 线性回归（MSE）：  
  \[
  L = \frac{1}{2n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2
  \]
- 逻辑回归（二元交叉熵）：  
  \[
  L = -\frac{1}{n} \sum_{i=1}^{n} [y_i \log(\hat{y}_i) + (1-y_i) \log(1-\hat{y}_i)]
  \]

### 2. 为什么 softmax 回归要用交叉熵损失，而不是 MSE？

1. **梯度优化特性**  
   交叉熵与 Softmax 结合时，梯度形式为 \( \hat{y} - y \)，不会因预测接近 0 或 1 而产生梯度饱和。  
   MSE + Softmax 会导致梯度消失（因为 Softmax 导数极小）。

2. **概率意义**  
   交叉熵衡量真实分布与预测分布的差异，符合最大似然估计原理；MSE 假设高斯噪声，不适用于分类问题。

3. **非凸优化**  
   MSE + Softmax 的损失函数非凸，容易陷入局部最优；交叉熵在 softmax 下是凸的（相对于 logits）。

### 3. 梯度下降的几种变体（BGD, SGD, Mini-batch GD）各有什么优缺点？

| 方法 | 优点 | 缺点 |
|------|------|------|
| **BGD**<br>（批量梯度下降） | 稳定收敛，梯度准确 | 大数据下计算慢，内存大，无法在线学习 |
| **SGD**<br>（随机梯度下降） | 更新快，可在线学习，有助于跳出局部极小 | 梯度噪声大，收敛不稳定，需精细调学习率 |
| **Mini-batch GD** | 折中方案，利用矩阵加速，噪声可控，稳定收敛 | 需选择 batch size，仍可能震荡 |

---

### 4. 反向传播中梯度消失和梯度爆炸的原因及解决方法

**原因**：
- **梯度消失**：深层网络使用 Sigmoid/Tanh 时，导数 <1，连乘后梯度指数衰减；或初始权重过小。
- **梯度爆炸**：权重初始化过大，连乘后梯度指数增长；或深层网络 ReLU 不当使用导致数值膨胀。

**解决方法**：
- 使用 **ReLU / LeakyReLU / Swish** 等激活函数
- 权重初始化：Xavier、He 初始化
- **Batch Normalization**
- **梯度裁剪**（gradient clipping）
- 使用 **残差结构（ResNet）**
- 适当降低网络深度或调整学习率

### 1. 手写 PyTorch 的 DataLoader 数据加载过程（不考虑高级 API）

```python
import numpy as np

class SimpleDataLoader:
    def __init__(self, X, y, batch_size=32, shuffle=True):
        self.X = X
        self.y = y
        self.batch_size = batch_size
        self.shuffle = shuffle
        self.n_samples = X.shape[0]
        self.indices = np.arange(self.n_samples)

    def __iter__(self):
        if self.shuffle:
            np.random.shuffle(self.indices)
        self.current_idx = 0
        return self

    def __next__(self):
        if self.current_idx >= self.n_samples:
            raise StopIteration
        start = self.current_idx
        end = min(self.current_idx + self.batch_size, self.n_samples)
        batch_indices = self.indices[start:end]
        self.current_idx = end
        return self.X[batch_indices], self.y[batch_indices]

# 使用示例
X = np.random.randn(100, 10)
y = np.random.randint(0, 2, (100, 1))
loader = SimpleDataLoader(X, y, batch_size=16)
for Xb, yb in loader:
    print(Xb.shape, yb.shape)
```

### 2. 如何用 PyTorch 实现一个自定义的自动求导函数（继承 `Function`）

```python
import torch
from torch.autograd import Function

class MyReLU(Function):
    @staticmethod
    def forward(ctx, input):
        ctx.save_for_backward(input)
        return input.clamp(min=0)

    @staticmethod
    def backward(ctx, grad_output):
        input, = ctx.saved_tensors
        grad_input = grad_output.clone()
        grad_input[input < 0] = 0
        return grad_input

# 使用
x = torch.tensor([-1., 2., 3.], requires_grad=True)
y = MyReLU.apply(x)
y.sum().backward()
print(x.grad)  # 输出 [0., 1., 1.]
```

### 3. 解释 `with torch.no_grad():` 的作用和使用场景

**作用**：
- 禁止 PyTorch 追踪计算图，不保存中间变量的梯度信息，减少内存占用并加速计算。

**使用场景**：
1. **模型评估/推理**：不需要梯度时，如 `model.eval()` 中。
2. **参数更新（手动优化）**：如在对参数进行就地更新时避免构建计算图。
3. **计算验证集指标**。
4. **生成对抗网络中的判别器/生成器交替更新时，冻结对方网络**。

示例：
```python
with torch.no_grad():
    for param in model.parameters():
        param -= lr * param.grad
```

## 场景题

### 1. 假如训练时 loss 一直不下降，你会从哪些方面排查？

**数据层面**：
- 数据是否打乱？标签是否错误？
- 输入是否归一化？
- 是否存在数据泄漏？

**模型层面**：
- 模型是否太简单/太复杂？
- 激活函数选择是否合适？
- 是否梯度消失/爆炸？（可打印各层梯度）

**训练配置**：
- 学习率过大（loss 震荡）或过小（几乎不下降）
- 优化器选择不当
- 损失函数与任务是否匹配

**代码实现**：
- 是否误用了 `no_grad` 导致参数未更新？
- 是否未调用 `zero_grad()` 导致梯度累加？

### 2. 如果你的模型在训练集上表现好，验证集上差，该怎么调整？

这是**过拟合**问题。

**解决方法**：
1. **增加数据量**：数据增强、收集更多数据。
2. **降低模型复杂度**：减少层数、神经元数。
3. **正则化**：
   - L2 正则（weight decay）
   - Dropout
   - Batch Normalization（也可视为隐式正则）
4. **早停（Early Stopping）**
5. **集成学习**：Bagging、模型集成
6. **简化特征**：减少特征维度或做特征选择
7. **交叉验证**：确保验证集划分合理
